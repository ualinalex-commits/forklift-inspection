# FORKLIFT PRE-USE INSPECTION APP — REBUILD BLUEPRINT

This document is a complete specification to rebuild the MEWP inspection app as a **forklift pre-use inspection app**. Every page, API route, database table, PDF coordinate, real-time subscription, auth mechanism, and environment variable is documented here. A developer with no prior knowledge of the original app can reconstruct the entire system from this file.

---

## 1. WHAT THE APP DOES

A public web app (no login for workers) for UK construction/warehouse sites to complete daily forklift pre-use inspections digitally, replacing paper forms.

**Flow:**
1. Site manager scans QR code → opens site dashboard at `/site/[siteId]`
2. Each forklift has an NFC tag / QR code with a unique URL
3. Worker scans tag → opens daily inspection form at `/check/[forkliftId]`
4. If already inspected today → shows "Already done" screen with details
5. Worker completes checklist (visual checks + function checks) → provides photo + signature → data saves to Supabase
6. PDF auto-generates after each inspection and also runs on a weekly schedule
7. Weekly PDF is stored in Supabase Storage and linked from the site dashboard

## 6. SUPABASE FUNCTIONS & VIEWS

### 6.1 RPC Functions

**`get_week_commencing(p_date date) → date`**
Returns the Monday of the week containing `p_date`.
```sql
CREATE OR REPLACE FUNCTION get_week_commencing(p_date date)
RETURNS date AS $$
  SELECT date_trunc('week', p_date::timestamp)::date;
$$ LANGUAGE sql IMMUTABLE;
```

**`get_or_create_weekly_sheet(p_forklift_id uuid, p_site_id uuid, p_machine_ref text, p_date date) → uuid`**
Finds the existing weekly sheet for the week containing `p_date`, or creates a new one. Returns the sheet UUID. Called client-side on inspection submit.
```sql
CREATE OR REPLACE FUNCTION get_or_create_weekly_sheet(
  p_forklift_id uuid,
  p_site_id     uuid,
  p_machine_ref text,
  p_date        date
) RETURNS uuid AS $$
DECLARE
  v_week_comm date;
  v_sheet_id  uuid;
BEGIN
  v_week_comm := get_week_commencing(p_date);
  SELECT id INTO v_sheet_id
    FROM weekly_inspection_sheets
   WHERE forklift_id = p_forklift_id
     AND week_commencing = v_week_comm;
  IF v_sheet_id IS NULL THEN
    INSERT INTO weekly_inspection_sheets
      (forklift_id, site_id, machine_ref, week_commencing, week_ending)
    VALUES
      (p_forklift_id, p_site_id, p_machine_ref, v_week_comm, v_week_comm + 6)
    RETURNING id INTO v_sheet_id;
  END IF;
  RETURN v_sheet_id;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;
```

### 6.2 Views

**`today_inspection_status`**
Joins forklifts + sites + today's entry + open defect count into one row per forklift. Used for quick status display.

**`weekly_operator_log`**
Pivots daily_inspection_entries so each row has the operator name, PAL card, and status for each day of the week, keyed by sheet_id and day_of_week.

**`weekly_sheet_summary`**
Pivots visual and function check results across 7 day columns. Each row = one item number. Columns: `mon_result`, `tue_result`, … `sun_result` for visual checks; `mon_ground_result`, `mon_platform_result`, … for function checks. Used by PDF generation to fill in the grid.

## 8. PAGES

### 8.1 `pages/check/[forkliftId].jsx` — Inspection Form

**Purpose:** The main worker-facing page. Workers scan an NFC/QR code that links here. No login required.

**URL:** `/check/[forkliftId]` where `forkliftId` is a UUID.

**State machine (pageStatus):**
- `loading` → fetching forklift record + today's entry from Supabase
- `not_found` → forklift UUID doesn't exist in DB
- `already_done` → a `daily_inspection_entries` row exists for today (show summary + any faults)
- `form` → ready for inspection (step 0–3)
- `submitting` → saving to DB
- `done` → success screen with fault summary
- `submit_error` → DB error, allow retry

**4-step form flow:**

**Step 0 — Operator Details:**
- Input: Full Name (required), Operator Card Number (optional), Forklift Owner Company (optional)
- Shows ThoroughExamCard (links to valid/expired thorough exam doc)
- "Start Visual Checks" button disabled until name is non-empty

**Step 1 — Visual Checks (items 1–20):**
- Each item: PASS / FAIL / N/A toggle (3-way)
- Grouped into sections with emoji headers
- Progress bar shows % completed
- Validation: all items must be answered before proceeding
- Unanswered items highlighted in orange on first "Next" attempt

**Step 2 — Function Checks (items 21–30):**
- Each item: two separate PASS/FAIL/N/A toggles — Ground (G) and Platform (P)
- Both G and P must be answered for each item
- Same validation and highlighting pattern as Step 1

**Step 3 — Review + Photo + Signature:**
- Summary table (machine, site, operator, PAL, date, visual count, function count, fault count)
- For each item marked FAIL: textarea to describe the fault in detail
- Photo section: `<input type="file" accept="image/*" capture="environment">` — takes camera photo
- Signature pad: `signature_pad` npm package, lazily imported via `import()` in useEffect when step === 3
  - Canvas sized to full width × 160px
  - Captures as PNG data URL → converts to Blob → uploads to Supabase Storage
- Both photo and signature are required — validation shows error and scrolls to missing element
- Submit button reads "⚠️ Submit with Faults" or "✅ Submit — All Clear"

**Submit logic (in order):**
1. Capture signature data URL and photo File BEFORE calling setPageStatus('submitting') — the canvas unmounts on re-render and sig pad is destroyed
2. Upload photo to `forklift-photos/{siteId}/{forkliftId}/{today}.jpg` (upsert: true)
3. Upload signature PNG to `signatures/{siteId}/{forkliftId}/{today}.png` (upsert: true)
4. Call `supabase.rpc('get_or_create_weekly_sheet', {...})` → get sheet UUID
5. Insert one row into `daily_inspection_entries` (includes `photo_url` and `signature_url`)
6. Insert 20 rows into `visual_check_results`
7. Insert 10 rows into `function_check_results`
8. If any FAILs: insert rows into `defect_log`
9. POST to `/api/trigger-pdf` with `{ forklift_id, sheet_id }` — fire and forget (awaited but result ignored)
10. setPageStatus('done')
11. On error: delete the entry row (rollback) and show error screen

**getServerSideProps:** Just extracts `params.forkliftId` and passes as prop. All data fetching is client-side.

### 8.2 `pages/site/[siteId].jsx` — Site Dashboard

**Purpose:** Manager-facing dashboard. Accessible by anyone with the link (public URL) but admin features (Add MEWP, Archive, Upload Exam) require auth or bypass token.

**URL:** `/site/[siteId]`

**On load:**
1. Check Supabase session → if logged in, fetch `user_profiles` → set `isAdmin` and `isMainAdmin`
2. If no session, check `localStorage.getItem(`site_bypass_${siteId}`)` === "granted" → set `isAdmin`
3. Check `localStorage.getItem("admin_bypass_token")` → set both `isAdmin` and `isMainAdmin`
4. Fetch site data, active forklifts, today's inspection entries, current week PDF URLs

**Components:**

**Stats panel:** 3 cards — Total MEWPs, Done Today (count of entries for today among active forklifts), Faults Today. All red/green colored.

**Progress bar:** Shows `doneToday / total` as a percentage. Turns green when 100%.

**Site QR Code:** Collapsible panel showing QR for the site URL. Download button renders QR to an offscreen canvas with site name text below and triggers download.

**ForkliftCard (active forklifts):** Collapsible card per forklift showing:

## 9. API ROUTES

### 9.1 `POST /api/trigger-pdf`

**Purpose:** Called immediately after each inspection submission. No auth — called client-side.

**Body:** `{ forklift_id: uuid, sheet_id: uuid }`

**Logic:**
1. If `sheet_id` given, look up `week_commencing` from `weekly_inspection_sheets`
2. If not, compute this week's Monday from today's date
3. Call `generateReport(forklift_id, weekCommencing)` synchronously
4. Return `{ ok: true }` or `{ error: ... }`

**Note:** This runs synchronously on Vercel. If PDF generation takes >10s the connection may time out on free plans, but the function will complete and the PDF URL will be updated in the DB. The site dashboard polls/subscribes to the sheet record so the link appears when ready.

---

### 9.2 `POST /api/generate-report`

**Purpose:** Same as trigger-pdf but requires `?secret=CRON_SECRET`. Used by the weekly cron job.

**Query params:** `secret=CRON_SECRET` (required — returns 401 if missing/wrong)

**Body:** `{ forklift_id: uuid, sheet_id?: uuid }`

**Logic:** Same as trigger-pdf but requires the secret param. Designed to be called by the Supabase Edge Function or Vercel cron.

---

### 9.3 `GET /api/admin/data`

**Auth:** Bearer token in Authorization header (Supabase JWT or ADMIN_BYPASS_TOKEN).

**Response:** `{ sites: [...], siteAdmins: [...] }` — all sites (including archived), all site_admin user_profiles with site names joined.

---

### 9.4 `POST /api/admin/add-site`

**Auth:** Bearer token (main_admin JWT or bypass token).

**Body:** `{ name, location, postcode, managerName }`

**Logic:** Inserts into `sites`, then updates `qr_code_url` with `${BASE_URL}/site/${site.id}`.

**Response:** `{ site: { ...newSiteRow, qr_code_url } }`

---

### 9.5 `POST /api/admin/create-site-admin`

**Auth:** Bearer token (main_admin only).

**Body:** `{ name, email, password, siteId }`

**Logic:**
1. `supabaseAdmin.auth.admin.createUser({ email, password, email_confirm: true })`
2. Insert into `user_profiles` with `role: "site_admin"`, `must_change_password: true`
3. If profile insert fails, delete the auth user (rollback)

**Response:** `{ siteAdmin: { ...profile, site_name } }`

---

### 9.6 `POST /api/admin/archive-site`

**Auth:** Bearer token (main_admin).

**Body:** `{ siteId: uuid, archive: boolean }`

**Logic:** Updates `sites.is_archived`.

---

### 9.7 `POST /api/admin/archive-site-admin`

**Auth:** Bearer token (main_admin).

**Body:** `{ adminId: uuid, archive: boolean }`

**Logic:** Updates `user_profiles.is_archived`.

---

### 9.8 `POST /api/admin/verify-bypass`

**No auth required.**

**Body:** `{ token: string }`

**Logic:** Compares `token` to `process.env.ADMIN_BYPASS_TOKEN`. Returns `{ valid: true/false }`.

---

### 9.9 `POST /api/site/verify-bypass`

**No auth required.**

**Body:** `{ token: string, siteId: uuid }`

**Logic:** Compares `token` to `process.env.ADMIN_BYPASS_TOKEN`. `siteId` must be present. Returns `{ valid: true/false }`.

---

### 9.10 `GET|POST /api/cron/weekly-archive`

**Purpose:** Weekly cron — generates PDFs for every active forklift that has a sheet for the current week.

**Auth:** `?secret=CRON_SECRET` query param (required).

**Config:** `export const config = { maxDuration: 300 }` — allows up to 5 minutes runtime on Vercel.

**Logic:**
1. Compute `weekCommencing` = Monday of today
2. Fetch all active non-archived forklifts
3. For each forklift: check if a `weekly_inspection_sheets` row exists for this week
4. If yes: call `generateReport(forklift.id, weekCommencing)`, record ok/error
5. If no: record as 'skipped'
6. Return summary `{ ok, skipped, errors, results }`

**Vercel cron setup** (in `vercel.json`):
```json
{
  "crons": [{
    "path": "/api/cron/weekly-archive?secret=CRON_SECRET_VALUE",
    "schedule": "0 17 * * 0"
  }]
}
```
Schedule `0 17 * * 0` = 17:00 UTC every Sunday (18:00 BST in summer).

**Alternative — Supabase Edge Function** (see Section 11).

### 10.3 `lib/generateReport.js`

The core PDF generation module. Exports `generateReport(forkliftId, weekCommencing)`.

**Steps:**
1. Fetch `weekly_inspection_sheets` row joined to `forklifts` → `sites` (machine ref, serial, site name)
2. Fetch `weekly_sheet_summary` view rows filtered by `sheet_id` → build `summaryByItem` lookup keyed by `item_number`
3. Fetch `weekly_operator_log` view rows filtered by `sheet_id` → build `operatorsByDay` lookup keyed by day abbreviation ('Mon', 'Tue', etc.)
4. Fetch `daily_inspection_entries` for the week range → override `operatorsByDay` (direct table rows are more reliable than view), build `mediaByDay` with `photo_url`, `signature_url`, `forklift_owner`
5. Fetch image bytes for each day's photo and signature in parallel (non-fatal: failures just skip the image)
6. Fetch `defect_log` rows for the week → build `defectsByDay` lookup keyed by day abbreviation
7. Call `buildPDF(...)` to produce a Buffer
8. Delete existing PDF from storage at `weekly-reports/{siteId}/{forkliftId}/{weekCommencing}.pdf`
9. Upload new PDF, get public URL
10. Update `weekly_inspection_sheets.pdf_url` and `pdf_generated_at`
11. Return the public URL

**`buildPDF({ forklift, sheet, summaryByItem, operatorsByDay, defectsByDay, mediaByDay })`:**
1. Read `public/template.pdf` from disk with `fs.readFileSync`
2. `PDFDocument.load(templateBytes)` via pdf-lib
3. Call `stripPlaceholders(pdfDoc)` — removes `{{KEY}}` placeholder text baked into the template (by decompressing content streams, blanking `{{...}}` inside BT...ET blocks, recompressing)
4. Embed Helvetica and HelveticaBold fonts
5. Stamp text onto Page 0 (landscape): machine ref at `(484, 559)`, week commencing at `(701, 559)`
6. Stamp visual check results 1–N: for each item, for each day, look up result from `summaryByItem[itemNum][`${day.lower}_result`]` → draw "PASS" (green), "FAIL" (red), "N/A" (gray) at the exact coordinate for that item/day intersection
7. Stamp function check results: for each item, for each day, draw Ground result and Platform result at separate y-offsets from the item's base y coordinate
8. Stamp initials on Page 1: for each day that has an entry, draw initials (`getInitials(operator_name)`) at the initials row
9. Remove Page 2 (the portrait "Daily Record" page baked into the template — replaced by the dynamic summary)
10. Call `appendDailySummaryPages(...)` — appends one or more A4 portrait pages with:
    - "DAILY INSPECTION SUMMARY" header bar
    - For each day that has an entry (in Mon–Sun order):
      - Dark header bar with day name + date
      - Info row: operator, PAL card, time, ALL CLEAR / FAULTS FOUND status
      - MEWP Owner row (if present)
      - Media row: photo (max 160×100px) + signature (max 130×55px) side by side
      - Fault table (if faults): columns for item #, description, fault details, status
11. Return `Buffer.from(await pdfDoc.save())`

### 10.4 `lib/createTemplate.js`

A one-off script that programmatically draws the inspection grid PDF using pdf-lib and writes it to `public/template.pdf`. Run it locally when the check item list changes.

**What it generates:**
- **Page 1 (A4 landscape, 841.92 × 595.32pt):**
  - Header band: title, "MACHINE: {{MACHINE_REF}}", "WEEK COMMENCING: {{WEEK_COMMENCING}}"
  - Column headers: day names (MONDAY…SUNDAY) with G/P sub-headers
  - VISUAL CHECKS section: rows for items 1–N with `{{MON_01}}`, `{{TUE_01}}`, etc. placeholders
  - FUNCTION CHECKS section: G and P sub-rows for items N+1–M with `{{MON_21_G}}`, `{{MON_21_P}}`, etc.
  - Bottom strip: "ALL FAULTS TO BE REPORTED TO SUPERVISOR" + initials row with `{{MON_INITIALS}}`, etc.

- **Page 2 (A4 portrait, 595.32 × 841.92pt):**
  - "MEWP/FORKLIFT DAILY RECORD OF INSPECTION" title
  - Daily Operator Log table (7 rows, columns: Day/Status/Operator/PAL/Date) with `{{MON_STATUS}}`, `{{MON_OPERATOR}}`, `{{MON_PAL}}` placeholders
  - Defect Log table (10 rows, columns: Item/Details/Date Noted/Date Reported/Engineer/Notes)
  - Weekly Supervisor Sign-Off section

This page is removed from the final output PDF; it's replaced by the dynamic `appendDailySummaryPages` output. It exists in the template only so the coordinates for Page 1 can be computed by walking the same layout algorithm.

**Coordinate system:** pdf-lib uses bottom-left origin. Higher `y` = higher on the page. Page 1 is 595.32pt tall, so the top is around y=583, the middle around y=297.

## 12. REAL-TIME SUBSCRIPTIONS

All subscriptions are set up in `pages/site/[siteId].jsx` via `useEffect`.

### Channel 1: `site-{siteId}-mewps`
```javascript
supabase.channel(`site-${siteId}-forklifts`)
  .on('postgres_changes',
    { event: 'UPDATE', schema: 'public', table: 'forklifts' },
    async (payload) => {
      // Only act if this event is for our site
      if (payload.new?.site_id !== siteId) return;
      // Re-fetch all active forklifts
      const { data } = await supabase.from('forklifts')
        .select('*').eq('site_id', siteId).eq('active', true)
        .or('is_archived.eq.false,is_archived.is.null')
        .order('created_at', { ascending: true });
      setForklifts(data || []);
      setStats(prev => ({ ...prev, total: (data || []).length }));
    }
  ).subscribe();
```
Note: INSERT events on forklifts are handled via local state (`onAdded` callback in AddForkliftModal). Real-time INSERT is not reliable when written via the service role key.

### Channel 2: `site-{siteId}-inspections`
```javascript
supabase.channel(`site-${siteId}-inspections`)
  .on('postgres_changes',
    { event: '*', schema: 'public', table: 'daily_inspection_entries',
      filter: `site_id=eq.${siteId}` },
    () => {
      refreshTodayData(siteId);
      setRtRefreshKey(k => k + 1);  // triggers WeeklyTracker reload in all cards
    }
  ).subscribe();
```
Listens for any change (INSERT/UPDATE/DELETE). Triggers a refresh of today's inspection data and increments `rtRefreshKey` which is passed to WeeklyTracker as a dependency.

### Channel 3: `site-{siteId}-sheets`
```javascript
supabase.channel(`site-${siteId}-sheets`)
  .on('postgres_changes',
    { event: 'UPDATE', schema: 'public', table: 'weekly_inspection_sheets',
      filter: `site_id=eq.${siteId}` },
    () => refreshPdfData()
  ).subscribe();
```
Listens for UPDATE on sheets (which happens when PDF is generated and `pdf_url` is written). Refreshes the PDF URL map.

### Polling fallback
A 10-second `setInterval` re-fetches forklifts, today's data, and PDF data as a backup in case real-time drops.

**Cleanup:** All channels are removed with `supabase.removeChannel(channel)` in the useEffect return function.

## 13. AUTH AND BYPASS TOKEN SYSTEM

### 13.1 Authentication Levels

| Level | Who | How they access |
|---|---|---|
| Workers | Anyone | No auth — just scan QR/NFC, land on `/check/[id]` |
| Site managers | Site-specific admin | Either Supabase login (site_admin role) or bypass link |
| Main admin | App owner | Either Supabase login (main_admin role) or bypass link |

### 13.2 Bypass Token Flow (for passwordless access)

**Admin bypass:**
1. Someone shares the link `https://yourapp.com/admin/bypass?token=ADMIN_BYPASS_TOKEN`
2. `pages/admin/bypass.jsx` sends `POST /api/admin/verify-bypass` with the token
3. API compares to `process.env.ADMIN_BYPASS_TOKEN` — returns `{ valid: true }`
4. Page stores `admin_bypass_token` in localStorage → redirects to `/admin`
5. `/admin` reads localStorage, sends the token as `Authorization: Bearer TOKEN` on all API calls

**Site bypass:**
1. Share `https://yourapp.com/site/bypass?token=ADMIN_BYPASS_TOKEN&site=SITE_UUID`
2. `pages/site/bypass.jsx` sends `POST /api/site/verify-bypass`
3. Returns `{ valid: true }` → stores `localStorage("site_bypass_{siteId}") = "granted"`
4. Redirects to `/site/{siteId}` where it finds the localStorage key and sets `isAdmin = true`

**Important:** The same `ADMIN_BYPASS_TOKEN` value is used for both admin and site bypass. The site bypass just checks that `siteId` is present in the request. There is no per-site token — the bypass grants admin access to all sites.

### 13.3 API Auth Pattern (all admin routes)

```javascript
async function verifyMainAdmin(req) {
  const token = req.headers.authorization?.split(" ")[1];
  if (!token) return null;
  // Check bypass token first
  if (process.env.ADMIN_BYPASS_TOKEN && token === process.env.ADMIN_BYPASS_TOKEN) {
    return { id: "bypass" };
  }
  // Check Supabase JWT
  const { data: { user }, error } = await supabaseAdmin.auth.getUser(token);
  if (error || !user) return null;
  const { data: profile } = await supabaseAdmin
    .from("user_profiles").select("role").eq("id", user.id).single();
  if (profile?.role !== "main_admin") return null;
  return user;
}
```

## 17. SIGNATURE PAD INTEGRATION

`signature_pad` is dynamically imported on Step 3 to avoid SSR issues:

```javascript
const canvasRef = useRef(null);
const sigPadRef = useRef(null);

useEffect(() => {
  if (step !== 3) return;
  let raf = requestAnimationFrame(() => {
    if (!canvasRef.current) return;
    import("signature_pad").then(({ default: SignaturePad }) => {
      const canvas = canvasRef.current;
      if (!canvas) return;
      canvas.width = canvas.offsetWidth || 320;
      canvas.height = canvas.offsetHeight || 160;
      sigPadRef.current = new SignaturePad(canvas, {
        backgroundColor: "rgb(255,255,255)",
        penColor: "#111827",
      });
    });
  });
  return () => {
    cancelAnimationFrame(raf);
    if (sigPadRef.current) { sigPadRef.current.off(); sigPadRef.current = null; }
  };
}, [step]);
```

**Critical:** Capture `sigPadRef.current.toDataURL("image/png")` BEFORE calling `setPageStatus("submitting")` — the canvas unmounts on re-render and the ref is nulled by the cleanup function.

**Converting data URL to uploadable Blob:**
```javascript
const base64 = sigDataUrl.split(",")[1];
const binary = atob(base64);
const bytes = new Uint8Array(binary.length);
for (let i = 0; i < binary.length; i++) bytes[i] = binary.charCodeAt(i);
const blob = new Blob([bytes], { type: "image/png" });
```

## 18. WEEKLY TRACKER COMPONENT

Shows Mon–Sun of the current week as colored squares. Green = inspected, Red = missed (past day with no inspection), Gray = future.

```javascript
function getWeekDates() {
  const today = new Date();
  const dow = today.getDay();
  const fromMon = (dow + 6) % 7;  // 0=Mon, 6=Sun offset
  const mon = new Date(today);
  mon.setDate(today.getDate() - fromMon);
  return Array.from({ length: 7 }, (_, i) => {
    const d = new Date(mon);
    d.setDate(mon.getDate() + i);
    return toLocalDateStr(d);
  });
}
```

Queries `daily_inspection_entries` filtered by `forklift_id` and `in(weekDates)`. Re-runs when `refreshKey` prop changes (incremented by real-time subscription).

## 22. SUPABASE RLS NOTES

The app uses a mix of anon-key (client) and service-role (server) access.

**Anon key can:**
- SELECT from `forklifts` where `active = true` and not archived (for inspection form)
- SELECT from `sites` (for site dashboard)
- SELECT/INSERT into `daily_inspection_entries` (one per forklift per day — UNIQUE constraint enforces)
- INSERT into `visual_check_results`, `function_check_results`, `defect_log`
- SELECT from `defect_log` (for displaying faults)
- SELECT from `weekly_inspection_sheets` (for PDF links)
- SELECT from `check_items`
- Upload to `forklift-photos`, `signatures`, `thorough-exams` buckets
- SELECT from all storage buckets (public read)

**Service role key (API routes only):**
- All of the above, plus:
- INSERT/UPDATE/DELETE on `sites`, `forklifts`, `user_profiles`
- CREATE auth users (`supabase.auth.admin.createUser`)
- Upload to `weekly-reports` bucket
- DELETE from `weekly-reports` (to replace PDF before uploading new version)

**Row Level Security policies needed:**
- `sites`: anon SELECT all active; service role all operations
- `forklifts`: anon SELECT where `active=true`; service role all; anon UPDATE for `thorough_exam_*` fields (if you want site dashboard to update without auth)
- `daily_inspection_entries`: anon INSERT + SELECT; service role all
- `visual_check_results`, `function_check_results`, `defect_log`: anon INSERT + SELECT; service role all
- `weekly_inspection_sheets`: anon SELECT; service role all
- `user_profiles`: anon SELECT own row only (for login redirect); service role all

The simplest approach for a low-security app: enable RLS but add permissive anon policies, relying on the UNIQUE constraints and UUID obscurity for data isolation rather than strict RLS.
